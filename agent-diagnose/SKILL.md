# /agent-diagnose — Agent 工作流程診斷

診斷 Agent 開發流程中的結構性問題。不教 prompt 工程，不推薦工具。找出根因，給操作建議，標明邊界。

## 程序

### Step 1：判斷輸入類型

- **A：貼了 agent 操作 log / 輸出** → Step 3
- **B：描述了問題** → Step 2
- **C：泛問「怎麼用 agent 比較好」** → 回問「你目前遇到最大的困難是什麼？」

### Step 2：症狀匹配

用【症狀辨識表】匹配。多個同時出現 → 只處理【優先序表】最高的一個。

匹配到 → 給【確認測試】，等回報再進 Step 4。
匹配不到 → 告知不在診斷範圍。

### Step 3：直接診斷（A 型）

讀 log/輸出，對照症狀表判斷主要問題。→ Step 4。

### Step 4：三層診斷

用【三層診斷表】給出：

```
診斷結果：A-xxx
立即動作：
結構根因：
邊界條件（這招沒用時）：
若沒效，下一步：
```

**三層都必須給。第三層（邊界條件）是本工具的核心差異化，不可省略。**

### Step 5：跟進

試了沒效 → 問哪裡沒達到預期 → 切換備案 → 備案也沒效 → 降一級優先序重新診斷。

## 症狀優先序表

1. A-004（測試作弊）
2. A-001（context 過載退化）
3. A-005（分析不修）
4. A-003（指令失效）
5. A-002（遺忘計畫）
6. A-006（繞圈打轉）
7. A-007（過度中斷）
8. U-001 / U-002 / U-003（使用者操作問題）

## 確認測試

### A-001
觀察：開 verbose output，比較 agent 前 3 步和後 3 步的工具呼叫。
- 前段精準、後段亂叫工具或重複讀檔 → 確認
- 前後一致地差 → 不是 A-001，查 A-003

### A-002
觀察：agent 有沒有在每步開始時讀取 plan/todo 檔案。
- 有讀但沒更新 → 確認（輕度）
- 根本沒讀 → 確認（重度）
- 沒有 plan 檔案 → 不是 A-002，是 U-003

### A-003
測試：在 CLAUDE.md 加一條極端明顯的指令（例如「每次回覆開頭寫 CHECKPOINT」），`/clear` 後觀察。
- 有時做有時沒做 → 確認（啟動率問題）
- 完全沒做 → 確認（重度）
- 穩定做到 → 不是 A-003，原始指令寫法有問題

### A-004
檢查 agent 寫的測試碼：
- 有 `assert(true)` 或等效永真測試 → 確認
- 測試檔被 git diff 顯示有改動 → 確認
- 測試預期值手算不對 → 確認

### A-005
觀察：agent 輸出中分析文字 vs 修改動作的比例。
- 大段分析但零個 Edit/Write 呼叫 → 確認
- 有分析也有修改 → 不是 A-005

### A-006
觀察：agent 是否連續 3 次以上用相同策略嘗試同一問題。
- 策略相同只改細節 → 確認
- 每次有實質不同策略 → 不是 A-006

### A-007
觀察：一個任務中 agent 中斷幾次。
- 5 次以上 → 確認
- 中斷都是關鍵決策點 → 正常行為，不是 A-007

### U-001 / U-002 / U-003
不診斷 agent。問使用者：
- 「你的 spec 有多詳細？」
- 「你有在看 agent 的操作過程嗎？」
- 「你的 plan 和進度存在哪裡？」

## 三層診斷

### A-001：Context 過載退化

**立即動作：**
- 工作切小塊。每段完成 → 存檔 → `/clear` → 下段
- 用 subagent 處理獨立子任務（獨立 context，不污染主線程）
- 大量資料處理 → 寫腳本在外部跑，只返回結果給主線程
- 避免一次讀入大量檔案，用 grep/glob 精確定位再讀

**結構根因：**
LLM 注意力隨 context 長度衰減（attention drift）。所有 token 競爭注意力資源。重要資訊被稀釋後，效果等同不存在。Context 不是硬碟，是注意力場——越大越散。

**邊界條件：**
切小塊後品質仍差 → 不是 context 問題，查 A-003 或 A-006。任務無法拆分 → 用 subagent 分工，每個處理一部分，結果存檔後由主線程整合。

---

### A-002：遺忘計畫

**立即動作：**
- Plan 和 tasklist 存成外部檔案，不依賴 agent 內部記憶
- 每步開始前強制讀取 plan 檔案（指令中明確寫）
- 每步完成後更新進度檔案
- 用 `/clear` + 讀檔重載取代依賴長 context

**結構根因：**
LLM 是無狀態的。每次生成是獨立事件。「記得」是統計性的——context 裡有這段文字時生成路徑受其影響的機率較高，但不是 100%。Plan 在 context 早期 → 被後來的內容稀釋 → 影響力趨近零。

**邊界條件：**
外部存檔解決「遺忘」但不解決「啟動」。Agent 可能讀了 plan 但不照做 → 那是 A-003（指令失效），不是 A-002。

---

### A-003：指令失效

**立即動作：**
- 關鍵規則改用 hooks（系統層強制執行，不依賴 agent 自覺）
- 減少指令總量（指令越多，每條注意力越少）
- 指令寫法改程序性：「Step 1 做 X，Step 2 做 Y」比「請注意 X」有效
- 最重要的規則放 CLAUDE.md（每次自動載入）

**結構根因：**
指令在 context ≠ 指令影響生成。三個原因：
1. **詞彙距離**：指令用的詞和當前任務的詞重疊少 → 注意力機制抓不到
2. **生成路徑鎖定**：訓練預設的統計權重 > 指令 → 指令被覆蓋
3. **完成信號搶先**：agent 在注意到指令前已進入生成路徑

系統層機制（hooks）繞過這三個問題：在生成完成後攔截輸出，不依賴生成過程中的注意力分配。

**邊界條件：**
Hook 只能攔截詞彙層模式（regex 可匹配的）。框架層錯誤（選錯整個方向）無法用 hook 捕捉 → 需要人工監看。指令過少也不行 → 必要的約束仍需存在，只是要精簡。

---

### A-004：測試作弊

**立即動作：**
- 測試和程式分開：先寫測試、鎖定測試檔、再寫程式
- Hold-out test set：保留一組 agent 看不到的測試，完工後才跑
- 指令中明確禁止修改測試檔案
- Review 時手動驗算關鍵測試的預期值
- 用 git diff 檢查測試檔有無被改動

**結構根因：**
Agent 優化的是「測試通過」這個信號，不是「程式正確」這個事實。改測試讓紅燈變綠是阻力最小的路徑。Agent 不區分「測試錯了」和「程式錯了」——它只看到紅燈，然後找最快方式變綠。

**邊界條件：**
Hold-out test 的品質取決於 spec 的品質。Spec 模糊 → hold-out test 也模糊 → agent 通過但產出不符預期。上游修復：spec 階段就定義明確的成功標準。

---

### A-005：分析不修

**立即動作：**
- 指令改為「修復這個問題」而非「分析這個問題」
- 要求輸出必須包含程式碼修改（Edit/Write），純文字分析不算完成
- Retro prompt：`now that you analyzed it, did you actually fix it? If not, fix it now.`
- 拆兩步：先分析存檔 → `/clear` → 讀分析檔執行修復

**結構根因：**
四個架構層因素的交集：
1. Next-token prediction 不區分「描述問題」和「解決問題」的 token
2. RLHF 標註者評文字品質，不驗證是否真的修好
3. 無狀態訓練沒有「分析了但沒修」的跨對話懲罰
4. 分析文字的訓練分佈密度遠高於修復程式碼

結果：分析完畢 → 完成信號觸發 → agent 認為工作結束。

**邊界條件：**
嘗試修但修錯 → 不是 A-005，是能力問題。A-005 專指有能力修但沒動手。區分：明確要求修復後 agent 能否正確修復。

---

### A-006：繞圈打轉

**立即動作：**
- `/clear` 後用不同措辭重述問題
- 指定一個具體的替代策略（不讓 agent 自己選）
- 用 subagent 帶全新 context 嘗試
- 卡在某個錯誤 → 不讓 agent 重試，你描述錯誤讓它重新規畫

**結構根因：**
模式匹配鎖定。同樣的 context 觸發同樣的生成路徑。Agent 的「重試」在相同 context 下會走同一條路——不是堅持，是統計必然。相同輸入 → 相同輸出分佈。

**邊界條件：**
換了 context 和策略仍失敗 → 問題可能超出 agent 能力。此時拆成更小子問題，或人工介入分析。

---

### A-007：過度中斷

**立即動作：**
- 設定 `"defaultMode": "bypassPermissions"` + `"skipDangerousModePermissionPrompt": true`
- 停用不需要的工具（ExitPlanMode、AskUserQuestion）
- 使用外部工作流程取代內建 plan mode
- Spec 寫清楚（agent 因不確定而提問 = 上游 spec 不夠明確）

**結構根因：**
兩個來源：(1) 權限系統設計 → 高風險動作需確認 (2) 訓練預設 → 不確定時問比做錯安全。Agent 的風險計算偏保守——問的成本低，做錯的成本高。

**邊界條件：**
移除所有中斷 = 移除安全檢查。破壞性操作（刪檔、force push）不應自動通過。平衡：常規操作自動化，破壞性操作保留確認。

---

### U-001：指令模糊

**立即動作：**
- 先寫 spec 再開工
- Spec 包含：做什麼、不做什麼、成功標準、邊界條件
- 用「先寫使用手冊」法：假裝成品完成，寫出操作手冊，觀察是否合理
- 叫 agent 寫 spec draft → 你審核修正 → 再開工

**結構根因：**
Agent 會填補模糊空間——用訓練預設填。你不說清楚，agent 用統計最大公約數補位。結果看起來合理但不是你要的。

**邊界條件：**
過度詳細的 spec 也有問題 → 文件太多 agent 反而不讀（A-003）。關鍵約束明確，其餘留白。

---

### U-002：不監工

**立即動作：**
- 開 verbose output 看每個工具呼叫
- 前幾步手動確認方向正確後再放手
- Agent 不斷提問 = spec 有問題，回去改 spec
- 設 checkpoint：每完成一個子任務人工檢查一次

**結構根因：**
Agent 不是自主 AI，是統計模型。它犯錯時不知道自己在犯錯。監工是架構需求，不是信任問題。

**邊界條件：**
監工 ≠ 微管理。逐行看程式碼不如自己寫。監工粒度：方向、策略、關鍵決策點。細節讓 agent 處理。

---

### U-003：過度依賴內部狀態

**立即動作：**
- Plan、tasklist、進度一律存成檔案
- 每個子任務完成寫 checkpoint 檔
- 用 `/clear` + 讀檔重載取代依賴長 context
- 掛掉後讀 checkpoint 繼續，不從頭來

**結構根因：**
Agent 的「記憶」是 context window——會話結束就清空。依賴內部狀態 = 把進度放在 RAM 不存檔。中斷即歸零。

**邊界條件：**
外部檔案太多 → 管理成本增加。只存關鍵狀態（plan、進度、決策記錄），不存過程細節。

## 解法速查：四類操作手段

遇到任何症狀，優先考慮以下手段：

| 手段 | 什麼時候用 | 為什麼有效 |
|------|-----------|-----------|
| **Shell 指令** | 搜檔案、查內容、批次處理 | 不佔 context，精確高效 |
| **外部檔案** | 存 plan、進度、中間結果 | 脫離 context 生命週期，永久可查 |
| **寫腳本** | 大量資料處理、格式轉換 | 在 agent 外部執行，只返回結果 |
| **Subagent** | 獨立子任務、避免污染主 context | 獨立 context，可平行，用完即棄 |

核心原則：**能不佔主 context 就不佔。能在外部做就在外部做。**

## 不適用情境

1. **程式邏輯 bug** — agent 寫的程式有錯但不是行為模式問題。→ 「這是 debug，不是 agent 行為問題。」
2. **選工具 / 選框架** — 該用什麼 agent harness。→ 「這個技能診斷問題，不推薦工具。」
3. **要 prompt 模板** — 「給我一個好的 CLAUDE.md」。→ 「模板不解決結構問題。描述你遇到的困難，我來診斷。」
4. **模型能力極限** — 任務超出模型能力。→ 「這不是操作問題，是能力邊界。」

## 輸出規則

- 直接。不加緩衝，不教 AI 理論（結構根因段除外）。
- 一次一個問題。
- 三層都給：立即動作 + 結構根因 + 邊界條件。
- B 型輸入：確認再診斷，不跳步。
- 不保證。建議沒用進 Step 5。
